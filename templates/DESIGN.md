# テンプレート作成 詳細設計書

## 概要

本ドキュメントは、CSチームのテンプレート整備プロジェクトの詳細設計を記載します。

**目標**: 「配って成果に貢献できる」処方箋をつくる

---

## 1. テンプレート一覧と優先順位

### 1.1 優先度マトリクス

| 優先度 | テンプレート | 目的 | 担当 | 完了目標 |
|--------|-------------|------|------|---------|
| **P1** | ヒアリングシート | 営業→CS引継ぎ時の情報収集を標準化 | 嘉生→糸谷 | 2月上旬 |
| **P1** | 配信設定（3パターン） | ライブ/疑似ライブ/オンデマンドの設定手順標準化 | 鈴木 | 2月上旬 |
| **P2** | キャンペーンページ（2パターン） | セミナー集客・製品紹介ページの雛形 | 鈴木 | 2月中旬 |
| **P2** | セミナー構成 | 本数・時間配分・アンケート・ステップメール設計 | 鈴木 | 2月中旬 |
| **P3** | QBRテンプレート | 定期検診用のアジェンダと準備物リスト | 嘉生 | 2月末 |
| **P3** | サムネイル（3パターン） | 配信サムネイルのデザイン雛形 | 鈴木・大本 | 2月下旬 |
| **P4** | Salesforceレポート・通知 | ヘルススコア・予兆検知用のレポート設定 | 津田 | 3月上旬 |

### 1.2 優先順位の根拠

**P1（最優先）の理由：**
- **ヒアリングシート**: 2/1担当制開始に向けて、営業からCSへの情報引継ぎを標準化する必要がある
- **配信設定**: オンボーディングの核となるテンプレート。顧客の初回配信成功率に直結

**P2（中優先）の理由：**
- **キャンペーンページ/セミナー構成**: 顧客の成果創出（パイプライン転換率向上）に直接貢献

**P3/P4の理由：**
- **QBR/サムネイル**: S/Aセグメント向けの高付加価値施策。基盤ができてから整備
- **SFレポート**: ヘルススコアのSF実装完了後に作成

## 2. テンプレートの形式選定

### 2.1 形式選定マトリクス

| テンプレート | 推奨形式 | 理由 |
|-------------|---------|------|
| **ヒアリングシート** | Googleフォーム + スプレッドシート | 営業が入力しやすく、SFへのインポートが容易 |
| **配信設定** | Markdown + PDF | 手順書として閲覧性が高く、バージョン管理可能 |
| **キャンペーンページ** | HTML/CSS テンプレート | ネクプロ上で直接利用可能 |
| **セミナー構成** | スプレッドシート | 時間配分・項目の編集が容易 |
| **QBRテンプレート** | PowerPoint（既存システム活用） | 顧客向け資料として提示するため、見栄えが重要 |
| **サムネイル** | Canva / PowerPoint | デザイン編集が容易、非デザイナーでも使える |
| **SFレポート・通知** | Salesforceネイティブ | SF上で直接利用 |

### 2.2 既存PowerPointシステム活用

**`src/core/templates.py`を活用すべきテンプレート：**
- **QBRテンプレート**: `quarterly_review`プレゼンテーションタイプをベースにカスタマイズ

## 4. 各テンプレートの内容設計

### 4.1 ヒアリングシート

**目的**: 営業からCSへの引継ぎ時に必要な情報を漏れなく収集

**フィールド設計**:

#### 基本情報
- 企業名（必須）
- 契約開始日（必須）
- 契約金額（必須）
- 契約期間

#### 利用目的
- 主な利用目的（選択）: セミナー集客 / 社内研修 / 採用 / IR / その他
- 想定配信頻度: 月○回
- 想定視聴者数: ○名

#### 配信形式
- 希望配信形式（複数選択可）: ライブ / 疑似ライブ / オンデマンド
- 過去のウェビナー経験: あり / なし
- 利用ツール経験: Zoom / Teams / その他

#### 技術要件
- メディアサイト利用希望: あり / なし
- Salesforce連携希望: あり / なし
- SSO連携希望: あり / なし

#### 顧客側体制
- 主担当者名
- 主担当者メール
- 意思決定者名
- IT部門の関与度: 高 / 中 / 低

#### 営業からの申し送り事項
- 特記事項（フリーテキスト）
- 導入の決め手
- 懸念事項・要注意点

### 4.4 セミナー構成テンプレート

#### タイムライン雛形（60分構成）

| セクション | 時間（分） | 内容 | 担当 | 資料 |
|-----------|-----------|------|------|------|
| オープニング | 5 | 挨拶、本日のアジェンダ | 司会 | スライド1-3 |
| 本編1 | 20 | [トピック] | 講師 | スライド4-15 |
| 休憩/質問受付 | 5 | チャット質問確認 | 司会 | - |
| 本編2 | 20 | [トピック] | 講師 | スライド16-25 |
| Q&A | 5 | 質疑応答 | 全員 | - |
| クロージング | 5 | まとめ、CTA | 司会 | スライド26-28 |

#### アンケートテンプレート（5-7問）
1. 本セミナーの満足度（5段階）
2. 内容の理解度（5段階）
3. 最も参考になった内容（選択式）
4. 今後聞きたいテーマ（複数選択）
5. 商談希望（はい/いいえ）
6. その他ご意見（自由記述）

### 4.5 QBRテンプレート

#### PowerPoint生成システムへの統合

`src/core/templates.py`の`PRESENTATION_TYPES`に追加：

```python
"qbr_customer_review": {
    "name": "QBR - 顧客定期レビュー",
    "description": "四半期ビジネスレビュー用テンプレート",
    "recommended_theme": "corporate",
    "slide_structure": [
        {"type": SlideType.TITLE, "purpose": "QBR表紙"},
        {"type": SlideType.AGENDA, "purpose": "本日のアジェンダ"},
        {"type": SlideType.CHART, "purpose": "利用状況サマリー"},
        {"type": SlideType.TWO_COLUMN, "purpose": "成果と課題"},
        {"type": SlideType.CONTENT, "purpose": "次四半期の目標"},
        {"type": SlideType.TIMELINE, "purpose": "アクションプラン"},
        {"type": SlideType.QA, "purpose": "質疑応答"}
    ]
}
```

#### QBRアジェンダ（60分構成）

| 時間 | 内容 | 医療比喩 |
|------|------|---------|
| 0-5分 | アイスブレイク | 挨拶・問診前の声かけ |
| 5-15分 | 前四半期の成果振り返り | 検査結果の説明 |
| 15-25分 | 課題ディスカッション | 問診 |
| 25-40分 | 次四半期のゴール設定 | 治療計画の策定 |
| 40-50分 | ネクプロからの提案 | 処方箋の発行 |
| 50-60分 | 質疑応答・次回日程 | 次回の検診予約 |

### 4.6 サムネイルテンプレート

**3パターン**:

1. **テキスト中心（シンプル）**
   - サイズ: 1280x720px
   - 背景: 単色またはグラデーション
   - テキスト: 大きなタイトル + サブタイトル
   - 用途: 社内研修、シンプルな情報発信

2. **写真+テキスト**
   - サイズ: 1280x720px
   - 背景: 登壇者写真または製品写真
   - テキスト: オーバーレイでタイトル
   - 用途: セミナー、ウェビナー告知

3. **グラフィック重視**
   - サイズ: 1280x720px
   - 背景: イラストまたはアイコン
   - テキスト: キャッチコピー
   - 用途: 製品紹介、イベント告知

### 4.7 Salesforceレポート・通知

#### レポート定義

1. **ヘルススコアダッシュボード**
   - スコア分布（円グラフ）
   - セグメント別平均スコア
   - 週次推移グラフ

2. **解約予兆レポート**
   - 60日以上ログインなし顧客リスト
   - 配信数前月比50%以下顧客リスト
   - ヘルススコア赤顧客リスト

#### Slack通知テンプレート

```

顧客名: [顧客名]
担当CS: [担当者名]
シグナル: [60日未ログイン / 配信数急減 / ...]
最終接触日: [日付]
ヘルススコア: [スコア]点

48時間以内にご対応ください
```

## 7. 品質管理

### レビュープロセス

1. 担当者がドラフト作成
2. 嘉生が内容レビュー
3. 1社で試験運用
4. フィードバック反映
5. CS全員に共有

### バージョン管理

- GitでMarkdown/HTML/CSSファイルをバージョン管理
- 変更履歴はコミットメッセージに記録
- 大きな変更はREADME.mdの「更新履歴」セクションに記載
