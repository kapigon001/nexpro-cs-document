## 2. 顧客管理（Who：誰にどう向き合うか）

### ロジックツリー

```
2. 顧客管理
│
├── 2.1 セグメンテーション
│   ├── S（Strategic）：年間500万以上/エンプラ → 5-8社/人
│   ├── A（Growth）：年間100-500万、成長余地大 → 10-15社/人
│   ├── B（Stable）：年間100万未満、安定利用 → 15-20社/人
│   └── C（Low-touch）：利用頻度低 → 20-30社/人
│
├── 2.2 ライフサイクル管理
│   ├── 導入期（0-3ヶ月）：オンボーディング
│   ├── 活用期（3-6ヶ月）：定着支援
│   ├── 拡大期（6-12ヶ月）：アップセル提案
│   ├── 成熟期（12ヶ月〜）：維持・深化
│   └── 更新期（更新2ヶ月前〜）：更新確認
│
└── 2.3 ヘルススコア
    ├── 利用状況（40点）：配信数、ログイン日数
    ├── エンゲージメント（30点）：接点頻度、MTG実施
    ├── 成果（20点）：パイプライン転換率
    └── 契約（10点）：更新までの日数
```

### 2.1 セグメンテーション詳細

| セグメント | 定義 | 担当上限 | 接点頻度 | 対応方針 |
|-----------|------|---------|---------|---------|
| **S** | 年間売上500万以上 or エンタープライズ | 5-8社/人 | 月1回以上 | ハイタッチ |
| **A** | 年間売上100-500万、成長余地大 | 10-15社/人 | 四半期1回以上 | ミドルタッチ |
| **B** | 年間売上100万未満、安定利用 | 15-20社/人 | 半年1回以上 | ロータッチ |
| **C** | 利用頻度低、セルフサービス中心 | 20-30社/人 | 年1回以上 | テックタッチ |

### 2.2 ライフサイクル詳細

```
契約 ─→ 導入期 ─→ 活用期 ─→ 拡大期 ─→ 成熟期 ─→ 更新期
        (0-3M)    (3-6M)    (6-12M)   (12M〜)   (更新2M前)
          │         │         │         │         │
          ▼         ▼         ▼         ▼         ▼
       オンボード  定着支援   拡大提案  維持深化   更新確認
       初回配信   改善提案   アップセル  QBR     成果サマリー
       成功      Tips共有   他部門展開  事例化   更新手続き
```

| フェーズ | 期間 | ゴール | 主要アクション |
|---------|------|-------|---------------|
| 導入期 | 0-3ヶ月 | 初回配信成功 | キックオフ、初期設定、初回配信支援 |
| 活用期 | 3-6ヶ月 | 定期配信定着 | 利用状況確認、改善提案、3ヶ月レビュー |
| 拡大期 | 6-12ヶ月 | 利用範囲拡大 | 6ヶ月レビュー、アップセル提案、他部門展開 |
| 成熟期 | 12ヶ月〜 | ロイヤル化 | QBR、事例化、紹介依頼 |
| 更新期 | 更新2ヶ月前〜 | 契約更新 | 更新意向確認、成果サマリー、更新手続き |

### 2.3 ヘルススコア詳細

| カテゴリ | 指標 | 配点 | 算出方法 |
|---------|------|------|---------|
| 利用状況 | 月間配信数 | 25点 | 0回=0点、1-2回=15点、3回以上=25点 |
| 利用状況 | 月間ログイン日数 | 15点 | 0日=0点、1-5日=8点、6日以上=15点 |
| エンゲージメント | 直近接点からの日数 | 20点 | 60日以上=0点、30-59日=10点、30日未満=20点 |
| エンゲージメント | MTG実施状況 | 10点 | 四半期MTGなし=0点、あり=10点 |
| 成果 | パイプライン転換率 | 20点 | 業界平均以下=5点、平均=12点、平均以上=20点 |
| 契約 | 更新までの日数 | 10点 | 3ヶ月以内=10点、3-6ヶ月=7点、6ヶ月以上=5点 |

| スコア | 状態 | アクション |
|--------|------|-----------|
| 80-100点 | 🟢 健全 | 通常対応、アップセル機会探索 |
| 60-79点 | 🟡 要注意 | 接点強化、課題ヒアリング |
| 40-59点 | 🟠 警戒 | 即時介入、改善プラン提示 |
| 0-39点 | 🔴 危険 | 緊急対応、マネージャーエスカレーション |

## 3. プロセス（How：どう支援するか）

### ロジックツリー

```
3. プロセス
│
├── 3.1 オンボーディング
│   ├── Step0: 事前設定（代理対応）
│   ├── Step1: キックオフMTG（ゴール設定）
│   ├── Step2: 動画視聴（機能学習）
│   ├── Step3: 初回配信前チェックMTG
│   └── Step4: 初回配信後振り返りMTG
│
├── 3.2 定期接点
│   ├── QBR（四半期ビジネスレビュー）：S/Aセグメント
│   ├── 月次メール：活用Tips配信
│   └── 随時：新機能案内
│
├── 3.3 解約防止
│   ├── 予兆検知：ヘルススコア🔴、60日未ログイン等
│   ├── 介入：48時間以内に接触
│   └── リカバリー：解約理由別プレイブック
│
└── 3.4 アップセル・クロスセル
    ├── 機会検知：利用量増加、新機能関心等
    ├── 提案：CSから提案 or 営業連携
    └── 対象：SD動画、コンサル、追加オプション
```

### 3.1 オンボーディング詳細

```
【新オンボーディングフロー】

契約 ──→ Step0 ──→ Step1 ──→ Step2 ──→ Step3 ──→ Step4 ──→ 自走
         事前設定    キックオフ  動画視聴   配信前     振り返り
         (代理)     (MTG)      (非同期)   チェック   (MTG)
                                          (MTG)
         │          │          │          │          │
         ▼          ▼          ▼          ▼          ▼
      ・初期設定   ・ゴール設定 ・機能学習  ・設定確認  ・成果確認
      ・アカウント ・運用相談   ・自己学習  ・最終調整  ・改善点
        発行                              ・Q&A      ・次ステップ
```

| ステップ | タイミング | 担当 | 内容 |
|---------|-----------|------|------|
| Step0 | 契約後〜キックオフ前 | CS | 初期設定を代理で完了 |
| Step1 | 契約後1週間以内 | CS+顧客 | ゴール設定、運用方針の相談 |
| Step2 | Step1後〜初回配信前 | 顧客 | 動画で機能学習（非同期） |
| Step3 | 初回配信1週間前 | CS+顧客 | 設定確認、最終調整、Q&A |
| Step4 | 初回配信後1週間以内 | CS+顧客 | 成果振り返り、改善点、次ステップ |

#### 定期メール配信

### 3.3 解約防止詳細

#### 解約予兆シグナル

| 予兆シグナル | 検知方法 | 重要度 |
|-------------|---------|-------|
| 60日以上ログインなし | SF自動検知 | 🔴 高 |
| 月間配信数が前月比50%以下 | SF自動検知 | 🟠 中 |
| 問い合わせ・クレームの増加 | サポート連携 | 🟠 中 |
| 担当者変更の連絡 | 顧客からの連絡 | 🟡 低-中 |
| 競合検討の情報 | 営業情報共有 | 🔴 高 |

#### 介入フロー

```
予兆検知 ──→ Slack通知 ──→ 48時間以内に接触 ──→ 状況ヒアリング
                                                    │
                    ┌───────────────────────────────┼───────────────────┐
                    ▼                               ▼                   ▼
               課題なし                          課題あり            解約意向あり
               →経過観察                        →改善提案           →リカバリー
                                                                    →MGRエスカレーション
```

#### リカバリープレイブック

| 解約理由 | リカバリーアクション |
|---------|---------------------|
| 施策終了・中止 | 別用途の提案（社内研修、採用等）、次期施策の相談 |
| コスト | ROI可視化、プラン見直し提案 |
| 機能不足 | 代替機能の提案、開発ロードマップ共有 |
| 担当者変更 | 新担当者へのオンボーディング再実施 |
| 競合乗り換え | 差別化ポイントの再訴求 |

## 5. リスク管理（If：問題発生時どうするか）

### ロジックツリー

```
5. リスク管理
│
├── 5.1 リスクシナリオ
│   ├── 高影響：Sセグメント解約、主要担当者離職
│   ├── 中影響：複数顧客同時解約、システム障害
│   └── 低影響：担当不在時の問い合わせ
│
└── 5.2 緊急対応フロー
    ├── Sセグメント解約リスク：即時MGRエスカレーション
    ├── システム障害：開発連携→顧客通知
    └── 担当不在：SF情報でチーム対応
```

### 5.1 リスクシナリオ詳細

| リスク | 発生確率 | 影響度 | 対応策 |
|-------|---------|-------|-------|
| Sセグメント顧客の解約 | 低 | 🔴高 | ヘルススコア監視、早期介入、役員エスカレーション |
| 主要担当者の突然の離職 | 中 | 🔴高 | SF情報集約、引継ぎドキュメント整備 |
| 複数顧客の同時解約 | 低 | 🔴高 | 解約予兆の早期検知、四半期レビュー |
| 新メンバーの立ち上げ遅延 | 中 | 🟡中 | オンボーディングチェックリスト、メンター制度 |
| システム障害時の顧客対応 | 中 | 🟡中 | 緊急連絡フロー、FAQ整備 |
| 担当不在時の問い合わせ | 高 | 🟢低 | SF情報共有、チームでの対応体制 |

### 5.2 緊急対応フロー詳細

#### Sセグメント解約リスク発生時

```
リスク検知 ──→ 担当CS即時報告 ──→ 嘉生エスカレーション ──→ 糸谷報告（必要時）
                                                              │
                                                              ▼
                                                    緊急対応MTG（24時間以内）
                                                              │
                                                              ▼
                                                    リカバリープラン実行
```

#### 担当変更時の引継ぎチェックリスト

- [ ] SF顧客情報が最新化されているか
- [ ] 直近の対応履歴・課題が記載されているか
- [ ] 顧客のキーパーソン情報があるか
- [ ] 進行中の案件・約束事項があるか
- [ ] 顧客特有の注意点があるか
- [ ] 新担当者への顧客紹介MTGを設定したか

## 付録：CSの範囲定義

### やること / やらないこと

| やること | やらないこと |
|---------|-------------|
| 顧客の成果創出に向けた伴走支援 | 技術的なバグ修正（→開発） |
| オンボーディング・活用支援 | 新規顧客への営業活動（→営業） |
| 解約防止のための能動的アプローチ | 請求・契約事務（→管理） |
| アップセル機会の発見・提案 | 24時間緊急対応（→サポート） |
| 顧客要望のプロダクトへのフィードバック | 個別カスタマイズ開発の実施 |
| 成功事例の収集・横展開 | マーケティングコンテンツの制作（→マーケ連携） |
